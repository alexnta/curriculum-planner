## Ground rules before Phase 1

Use these rules for the whole project:

1. **No real UI before the core logic is tested.**
   The only allowed early page is a health check or empty home page.

2. **Seed only a small subset first.**
   Do **not** type all 48 subjects on day one. Start with 8–12 subjects around the most useful chains:

   * PRF192
   * PRO192
   * CSD201
   * DBI202
   * PRJ301
   * HSF302
   * JPD113
   * JPD123
   * SWE202c
   * IOT102

3. **`termNo` is sorting info, not availability info.**
   Use it to order recommendations. Do **not** assume it means “offered this semester.”

4. **Use max-subjects first, not max-credits.**
   Your curriculum dump does not show credits. Add `credits` later only if you find official values.

5. **No login in v1.**
   Use one demo student first.

6. **Use these truth cases in tests from the beginning:**

   * `PRO192` passed → `CSD201` eligible
   * `PRO192` only → `PRJ301` **not** eligible
   * `PRO192 + DBI202` passed → `PRJ301` eligible
   * `PRJ301` passed → `HSF302` eligible

Assume your root package is something like:

```text
com.yourname.curriculumplanner
```

And the main folders are:

```text

src/main/java/.../model
src/main/java/.../core
src/main/java/.../service
src/main/java/.../dto
src/main/webapp/WEB-INF/views
src/test/java
sql
```

## Phase 3 — Pure prerequisite engine

### Goal

Build and test the **core prerequisite logic in plain Java**, with zero servlet or JSP dependency.

### Very important rule

A simple graph is useful for:

* chain traversal
* cycle detection
* visualization

But it is **not enough** for eligibility when OR-groups exist.

So:

* **Graph structure** handles traversal
* **Prerequisite groups** handle actual rule evaluation

Do **not** flatten OR logic into a plain edge list and then trust that for eligibility.

### Create these files

* `core/GraphBuilder.java`
* `core/PrerequisiteGraph.java`
* `core/PrerequisiteChainResolver.java`
* `core/CycleDetector.java`
* `core/EligibilityEngine.java`
* `dto/MissingRequirementDTO.java`
* `dto/EligibilityResultDTO.java`
* `src/test/java/.../core/GraphBuilderTest.java`
* `src/test/java/.../core/PrerequisiteChainResolverTest.java`
* `src/test/java/.../core/CycleDetectorTest.java`
* `src/test/java/.../core/EligibilityEngineTest.java`

### What these classes should do

* `GraphBuilder` builds adjacency data from prerequisite records
* `PrerequisiteChainResolver` finds all upstream prerequisites
* `CycleDetector` rejects bad prerequisite cycles
* `EligibilityEngine` answers:

  * eligible or not
  * which requirements are missing

### Definition of Done

* [ ] `PRO192` passed → `CSD201` eligible
* [ ] `PRO192` only → `PRJ301` not eligible
* [ ] missing requirement for that case includes `DBI202`
* [ ] `PRO192 + DBI202` passed → `PRJ301` eligible
* [ ] `PRJ301` passed → `HSF302` eligible
* [ ] An artificial cycle inserted in a unit test is detected
* [ ] `mvn test` passes for all core prerequisite tests
* [ ] `core` package has no servlet imports and no JDBC code

### Not allowed yet

* No semester planner
* No JSP
* No servlet controllers except health

## Phase 4 — Pure planner and validator engine

### Goal

Build the **next-subject recommendation** and **plan validation** logic, still in plain Java.

### Simplify the rules for v1

For version 1:

* only already-passed subjects count as satisfying prerequisites
* a selected subject in the same plan does **not** unlock another selected subject

Example:

* selecting `PRJ301` and `HSF302` together should still fail if `PRJ301` is not already passed

That keeps the system predictable and much easier to test.

### Create these files

* `core/RecommendationEngine.java`
* `core/PlanValidator.java`
* `dto/PlanRequestDTO.java`
* `dto/RecommendedSubjectDTO.java`
* `dto/BlockedSubjectDTO.java`
* `dto/PlanValidationResultDTO.java`
* `src/test/java/.../core/RecommendationEngineTest.java`
* `src/test/java/.../core/PlanValidatorTest.java`

### What the engine should do

* return all currently eligible subjects
* return blocked subjects with reasons
* sort recommendations by `termNo`, then `code`
* validate a selected plan against:

  * max subjects
  * prerequisite satisfaction
  * already passed subjects not being re-selected

### Definition of Done

* [ ] Recommendation engine returns eligible subjects for the demo student
* [ ] `PRJ301` appears only when `PRO192` and `DBI202` are already passed
* [ ] blocked subjects come with human-readable reasons
* [ ] validator rejects already-passed subjects
* [ ] validator rejects plans over the configured max-subject count
* [ ] validator rejects `HSF302` if `PRJ301` is not already passed
* [ ] all planner tests pass
* [ ] still no real MVC pages

### Not allowed yet

* No subject list UI
* No planner form UI
* No graph frontend
